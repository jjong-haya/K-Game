# 발표 데모 스크립트

이 문서는 발표 때 실제로 화면을 어떻게 보여주고 어떤 말을 붙일지 정리한 스크립트입니다.

## 1분 소개

K-Game은 오늘의 단어와 프롬프트 룸을 통해 AI 응답과 사용자 참여를 함께 보여주는 서비스입니다.  
프론트는 S3와 CloudFront로 배포하고, API는 EC2에서 인증과 비즈니스 로직을 담당하며, AI 작업은 Lambda로 분리했습니다.  
저장소는 GitHub 공개를 고려해 구조, 문서, 배포, 보안을 같이 정리했습니다.

## 발표 흐름

### 1. 첫 화면

- 서비스 이름과 목적을 먼저 말합니다.
- `Daily Word`, `Prompt Room`, `관리`, `문서`가 어디에 있는지 보여줍니다.

### 2. 로그인 흐름

- 게스트 로그인 또는 소셜 로그인 흐름을 짧게 보여줍니다.
- 세션 쿠키 기반 인증이라는 점을 설명합니다.

### 3. Daily Word

- 문제 확인, 질문 입력, 응답 확인 순서로 보여줍니다.
- 실시간 응답이 `prompt-engine(question_answer)` 한 번으로 생성된다는 점을 설명합니다.
- 힌트는 `prompt-hint`가 따로 생성하고, 서버는 정답 누출 시 재호출 방어를 한다고 설명합니다.

### 4. 프롬프트 룸

- 제안 또는 평가 흐름을 보여줍니다.
- 사용자 액션, 서버 저장, AI 평가가 분리되어 있다는 점을 강조합니다.

### 5. 문서와 구조

- `README.md`를 열어 아키텍처 다이어그램을 보여줍니다.
- `docs/evaluator-guide.md`, `docs/security-checklist.md`, `docs/release-checklist.md`를 보여주며 공개 저장소 관리를 설명합니다.
- `infra/terraform`와 `infra/scripts`를 열어 배포 구조를 설명합니다.

## 발표 때 꼭 말할 것

- 프론트, API, Lambda, DB의 책임 분리가 분명합니다.
- 오늘의 단어 실시간 응답은 단일 `question_answer` 단계로 설명할 수 있습니다.
- 세션 쿠키와 최소 공개 범위를 기준으로 보안을 정리했습니다.
- 발표자뿐 아니라 평가자도 읽기 쉬운 문서 구조를 따로 뒀습니다.

## 질문 대비 포인트

- 왜 EC2와 Lambda를 같이 썼는지 설명할 수 있어야 합니다.
- 왜 실시간 질문 응답과 힌트, 자동 생성을 나눴는지 설명할 수 있어야 합니다.
- 왜 RDS를 private 기준으로 뒀는지 설명할 수 있어야 합니다.
- 왜 비밀값과 산출물을 Git에서 제외했는지 설명할 수 있어야 합니다.
