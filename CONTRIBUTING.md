# 기여 가이드

K-Game은 GitHub 공개 저장소 기준으로 정리된 monorepo를 목표로 합니다.  
이 저장소에 기여할 때는 코드의 동작뿐 아니라, 구조와 문서와 공개 위생까지 함께 맞추는 것을 원칙으로 합니다.

## 브랜치 규칙

- 기본 기준 브랜치는 `main`입니다.
- 작업 브랜치는 목적이 바로 드러나게 짧고 명확하게 만듭니다.
- 추천 형식은 `feature/<범위>`, `fix/<범위>`, `docs/<범위>`, `chore/<범위>`입니다.
- 한 브랜치에는 한 가지 목적만 담는 것이 좋습니다.

## 커밋 규칙

- 커밋 메시지는 가능하면 Conventional Commits 형식을 따릅니다.
- 추천 형식은 `type: 요약`입니다.
- 예시는 `feat: 로그인 세션 쿠키 전환`, `fix: daily word 빈 상태 처리`, `docs: 배포 문서 보강`입니다.
- 의미 없는 누적 커밋 대신, 리뷰하기 쉬운 단위로 나눕니다.

## PR 규칙

- PR 하나는 하나의 주제만 다룹니다.
- PR 설명에는 변경 이유, 핵심 변경점, 확인한 테스트를 적습니다.
- UI 변경이 있으면 스크린샷이나 짧은 설명을 같이 남깁니다.
- API, 인프라, 보안 관련 변경은 영향 범위와 롤백 가능성도 같이 적습니다.
- 비밀정보, 개인 키, 토큰, `.env` 실파일은 절대 포함하지 않습니다.

## 작업 전 확인

- 변경하려는 파일이 이미 다른 작업에서 수정 중인지 확인합니다.
- 다른 사람 변경은 되돌리지 않습니다.
- 필요한 경우 먼저 문서나 이슈에서 방향을 합의한 뒤 수정합니다.

## 검증 기준

- 코드 변경 후에는 관련 테스트를 먼저 실행합니다.
- 구조 변경이 있으면 README와 문서 링크도 같이 정리합니다.
- 공개 저장소 기준으로 불필요한 산출물, 로그, 캐시 파일은 남기지 않습니다.

## 추천 흐름

1. 브랜치 생성
2. 코드 또는 문서 수정
3. 테스트 실행
4. README 또는 문서 보강
5. PR 생성
